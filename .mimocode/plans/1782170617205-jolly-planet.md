# Issue #20 — Pentester Assistant Engine

## Branch

`git checkout -b feature/issue-20-pentester-assistant-engine`

## Overview

Add a Pentester Assistant Engine that consumes all intelligence products and produces prioritized investigation guidance, safe next steps, validation checklists, investigative questions, and executive narrative. The engine is read-only guidance — it never executes exploitation, brute force, DoS, or destructive payloads.

### `engine.py` — PentesterAssistantEngine
```python
class PentesterAssistantEngine:
    def __init__(self) -> None:
        self.context_loader = ContextLoader()
        self.triage = TriageEngine()
        self.next_steps = NextStepsEngine()
        self.planner = InvestigationPlanner()
        self.checklists = ValidationChecklistGenerator()
        self.evidence = EvidenceReasoner()
        self.questions = QuestionGenerator()
        self.narrative = RiskNarrative()
        self.recommendations = RecommendationBuilder()
        self.report_builder = AssistantReportBuilder()
        self.findings_mapper = AssistantFindingsMapper()
        self.hackerone = HackerOneIntegration()

    def analyze_project(
        self, project_path: Path | str, target_url: str | None = None
    ) -> AssistantReport:
        # 1. Load context
        # 2. Triage priorities
        # 3. Generate next steps
        # 4. Build investigation plan
        # 5. Build validation checklists
        # 6. Run evidence reasoner on priorities
        # 7. Generate questions
        # 8. Build risk narrative
        # 9. Build recommendations
        # 10. HackerOne integration guidance
        # 11. Zero-day safe notes
        # 12. Build report, save JSON profiles + findings
        # Return AssistantReport
```

- `findings/*.json` (all scanner findings)

Graceful fallback: if file missing, skip that source. `total_sources_loaded` tracks how many were found.

### `triage_engine.py` — TriageEngine
Sorts all investigation areas by composite score:
```
score = (severity_weight * sev_score)
      + (priority_weight * priority_score)
      + (confidence_weight * confidence_score)
      + (attack_chain_weight * chain_score)
      + (exploitability_weight * exploit_score)
      + (kev_bonus if KEV)
      + (epss_weight * epss_score)
      + (business_impact_weight * impact_score)
      + (manual_validation_weight * validation_value)
```
Each priority gets: rank, title, category, reason, evidence_refs, next_steps.

### `next_steps_engine.py` — NextStepsEngine
Generates safe steps based on finding types. NEVER suggests:
- brute force, bypass, DoS, dump, data modification, unauthorized access

Always suggests:
- manual review, confirm auth, validate access boundaries, document evidence, test with authorized accounts

Template-based: maps finding categories to predefined safe step templates.

### `investigation_planner.py` — InvestigationPlanner
Creates tasks for 8 types:
1. Web Application Review
2. API Security Review
3. Auth Review
4. Business Logic Review
5. CVE Verification
6. Attack Chain Review
7. Zero-Day Hypothesis Review
8. Bug Bounty Submission Review

Each task: title, objective, evidence, steps, expected_outcome, safety_notes, priority, estimated_effort.

### `validation_checklists.py` — ValidationChecklistGenerator
Generates checklists for vulnerability types present in the context:
- BOLA: identify object ref, confirm auth, validate boundaries, do not access outside auth, record sanitized evidence
- XSS: confirm reflection context, determine encoding, avoid harmful scripts, capture sanitized evidence
- CVE: confirm product/version, confirm patch status, avoid exploit execution
- Zero-Day: review signals, confirm anomaly, reproduce safely, no confirmed claims
- Auth: review claims, test access controls, validate session management
- API: inventory endpoints, test authorization, check rate limiting

### `evidence_reasoner.py` — EvidenceReasoner
For each priority/recommendation, generates:
- recommendation text
- evidence list (specific endpoints, parameters, scores)
- reasoning paragraph explaining WHY

### `question_generator.py` — QuestionGenerator
Generates pentester questions from context:
- "Does this endpoint require authentication?"
- "Can two authorized users access each other's objects?"
- "Is the admin API reachable from the public internet?"
- "Are JWT claims validated server-side?"
- "Are source maps intentionally exposed?"
- etc.
Maps finding types to question templates.

### `risk_narrative.py` — RiskNarrative
Generates 1-paragraph executive narrative describing highest-risk areas, supported by evidence.

### `recommendations.py` — RecommendationBuilder
Combines outputs from triage, evidence reasoner, and checklists into `AssistantRecommendation` objects.

### `report_builder.py` — AssistantReportBuilder
Assembles all outputs into `AssistantReport`. Also generates standalone MD and HTML reports.

### `hackerone_integration.py` — HackerOneIntegration
For each bounty-suitable finding:
- Suggests manual validation steps
- Adds checklist items
- Recommends additional evidence
- Indicates if confidence is sufficient or needs review

### `findings_mapper.py` — AssistantFindingsMapper
Maps assistant priorities to `FindingModel` for pipeline integration.

## 6. Safety Architecture

`PentesterAssistantEngine` enforces safety at multiple levels:

1. **NextStepsEngine**: Hardcoded deny list — never generates steps containing brute force, bypass, DoS, dump, destructive, unauthorized
2. **ValidationChecklists**: Each checklist item includes safety notes
3. **Zero-Day Integration**: Hypotheses always labeled "Research opportunity", "Manual validation required", "Hypothesis only" — never "confirmed"
4. **HackerOne Integration**: Recommendations include confidence assessment — "sufficient" or "needs review"
5. **Report disclaimer**: Every assistant report includes:
   > "The Pentester Assistant provides guidance for authorized manual review only. It does not confirm exploitation or replace professional judgment."
6. **findings_mapper**: Findings use INFO/LOW severity with category "pentest_guidance" — never claims confirmed vulnerability

## 10. Key Design Decisions

1. **Standalone Engine pattern** (like attack_chain, zero_day) — no ScannerBase inheritance
2. **Filesystem JSON I/O** — reads from profiles/findings, writes to profiles/assistant/
3. **Composite scoring** in triage — weighted formula with configurable weights
4. **Template-based** next steps, checklists, questions — deterministic, no randomness
5. **Safety-first** — deny lists, disclaimers, INFO-severity findings, no exploit claims
6. **Graceful degradation** — missing sources are skipped, not fatal
7. **HackerOne integration** is advisory only — suggests validation steps, doesn't auto-generate submissions
8. **Zero-day hypotheses** always labeled as research opportunities with "Hypothesis only" / "Manual validation required"
